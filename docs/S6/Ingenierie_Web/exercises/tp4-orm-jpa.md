# TP4 : ORM et JPA

> D'apres les consignes de l'enseignant dans : `S6/Ingenierie_Web/data/moodle/tp/tp4_orm_jpa/README.md`

Ce TP couvre le mapping objet-relationnel (ORM) avec JPA (Java Persistence API). Le projet modelise un album de cartes de joueurs de baseball.

Technologies : Java 11, JPA 2.1, Hibernate 5, HSQLDB (base en memoire), Maven, JUnit 5

Le projet se trouve dans `jpa/example/`. Un projet vide `jpa/empty-project/` est fourni pour demarrer.

---

## Modele de donnees

```
ModelElement (@MappedSuperclass, id auto-genere)
    |
    +-- Player (@Entity, @Inheritance JOINED, name)
    |     +-- BaseballPlayer (@Entity, totalHomeRuns, position)
    |
    +-- Album (@Entity, @OneToMany cards)
    |
    +-- PlayerCard (@Entity, @ManyToOne album, @ManyToOne player, @Embedded frontPicture, backPicture)

Picture (@Embeddable, pic)
Position (enum: PITCHER, CATCHER)
PositionConverter (@Converter)
```

## Exercice 1 : Configuration JPA (persistence.xml)

### Q1 : Comprendre la configuration

**Reponse :**

```xml
<!-- Fichier : src/main/resources/META-INF/persistence.xml (fourni) -->
<?xml version="1.0" encoding="UTF-8"?>
<persistence xmlns="http://xmlns.jcp.org/xml/ns/persistence" version="2.1">
    <persistence-unit name="playerCardDB" transaction-type="RESOURCE_LOCAL">
        <properties>
            <!-- HSQLDB en memoire (pas de persistence entre les redemarrages) -->
            <property name="javax.persistence.jdbc.driver" value="org.hsqldb.jdbcDriver"/>
            <property name="javax.persistence.jdbc.url" value="jdbc:hsqldb:mem:testdb"/>

            <!-- Recreer les tables a chaque demarrage -->
            <property name="javax.persistence.schema-generation.database.action"
                      value="drop-and-create"/>

            <!-- Generer les scripts DDL -->
            <property name="javax.persistence.schema-generation.scripts.action"
                      value="drop-and-create"/>
            <property name="javax.persistence.schema-generation.scripts.create-target"
                      value="create.ddl"/>
            <property name="javax.persistence.schema-generation.scripts.drop-target"
                      value="drop.ddl"/>

            <property name="hibernate.dialect" value="org.hibernate.dialect.HSQLDialect"/>
            <property name="hibernate.hbm2ddl.auto" value="create"/>
        </properties>
    </persistence-unit>
</persistence>
```

`drop-and-create` : les tables sont supprimees et recreees a chaque execution. Ideal pour les tests, pas pour la production.

## Exercice 2 : ModelElement -- Superclasse avec @Id

### Q2 : Ecrire la superclasse abstraite

**Reponse :**

```java
// Fichier : model/ModelElement.java (fourni)
package fr.insa.rennes.web.model;

import javax.persistence.GeneratedValue;
import javax.persistence.Id;
import javax.persistence.MappedSuperclass;

@MappedSuperclass     // PAS une table propre, attributs herites par les sous-classes
public abstract class ModelElement {
    @Id                // Cle primaire
    @GeneratedValue    // Valeur generee automatiquement par JPA
    protected int id;

    protected ModelElement() { super(); }

    public int getId() { return id; }
    protected void setId(final int id) { this.id = id; }
}
```

Points essentiels :
- `@MappedSuperclass` : ModelElement n'a PAS de table en base. Ses attributs (id) sont integres dans les tables des sous-classes.
- `@Id` + `@GeneratedValue` : JPA genere automatiquement la valeur de l'identifiant lors du `persist()`.
- Le constructeur `protected` est obligatoire pour JPA (utilise par reflexion).

## Exercice 3 : Player -- Entite avec heritage et requete nommee (NamedQuery)

### Q3 : Ecrire l'entite Player

**Reponse :**

```java
// Fichier : model/Player.java (fourni)
package fr.insa.rennes.web.model;

import javax.persistence.*;

@Entity
@Inheritance(strategy = InheritanceType.JOINED)   // Heritage : 1 table par classe
@NamedQueries({
    @NamedQuery(
        name = "getPlayerWithName",
        query = "SELECT p FROM Player p WHERE p.name = :name"
    ),
})
public class Player extends ModelElement {
    @Basic(optional = false)                       // NOT NULL
    @Column(name = "P_NAME", nullable = false)     // Nom de colonne personnalise
    private String name;

    @Transient                                     // PAS persiste en base
    private boolean anAttributeNotToMakePersistent;

    protected Player() { super(); }                // Constructeur JPA obligatoire

    public Player(final String playerName) {
        super();
        name = playerName;
    }

    public String getName() { return name; }
    public void setName(final String name) { this.name = name; }
}
```

Schema de la table generee :
```sql noexec
CREATE TABLE PLAYER (
    ID INTEGER GENERATED BY DEFAULT AS IDENTITY NOT NULL,
    P_NAME VARCHAR(255) NOT NULL,
    PRIMARY KEY (ID)
)
```

## Exercice 4 : BaseballPlayer -- Heritage JOINED

### Q4 : Ecrire la sous-classe

**Reponse :**

```java
// Fichier : model/BaseballPlayer.java (fourni)
package fr.insa.rennes.web.model;

import javax.persistence.*;

@Entity
public class BaseballPlayer extends Player {
    private int totalHomeRuns;

    @Basic(optional = false)
    @Column(name = "POSITION", nullable = false)
    private Position position;

    protected BaseballPlayer() { super(); }

    public BaseballPlayer(final String name, final Position playerPosition, final int homeRuns) {
        super(name);
        this.totalHomeRuns = homeRuns;
        position = playerPosition;
    }

    public int getTotalHomeRuns() { return totalHomeRuns; }
}
```

Schema genere avec la strategie JOINED :
```sql noexec
-- Table PLAYER (attributs de Player + id)
CREATE TABLE PLAYER (
    ID INTEGER NOT NULL, P_NAME VARCHAR(255) NOT NULL, PRIMARY KEY (ID)
)

-- Table BASEBALLPLAYER (attributs specifiques + cle etrangere vers PLAYER)
CREATE TABLE BASEBALLPLAYER (
    ID INTEGER NOT NULL,          -- Cle etrangere vers PLAYER.ID
    TOTALHOMERUNS INTEGER,
    POSITION VARCHAR(255) NOT NULL,
    PRIMARY KEY (ID),
    FOREIGN KEY (ID) REFERENCES PLAYER(ID)
)
```

Les 3 strategies d'heritage JPA :

| Strategie | Tables generees | Avantages | Inconvenients |
|-----------|-----------------|-----------|---------------|
| `SINGLE_TABLE` | 1 table pour toute la hierarchie | Rapide, pas de jointures | Colonnes NULL |
| `JOINED` | 1 table par classe | Normalise, pas de NULL | Jointures couteuses |
| `TABLE_PER_CLASS` | 1 table par classe concrete | Pas de jointures | Duplication des colonnes |

## Exercice 5 : Position et PositionConverter

### Q5 : Ecrire le convertisseur enum -> chaine de caracteres

**Reponse :**

```java
// Fichier : model/Position.java (fourni)
public enum Position {
    PITCHER,
    CATCHER
}
```

```java
// Fichier : model/PositionConverter.java (fourni)
package fr.insa.rennes.web.model;

import javax.persistence.AttributeConverter;
import javax.persistence.Converter;

@Converter(autoApply = true)    // Applique automatiquement a tous les attributs Position
public class PositionConverter implements AttributeConverter<Position, String> {
    @Override
    public String convertToDatabaseColumn(final Position position) {
        if (position != null) {
            switch (position) {
                case CATCHER: return "C";
                case PITCHER: return "P";
            }
        }
        return "";
    }

    @Override
    public Position convertToEntityAttribute(final String posStr) {
        if (posStr != null) {
            switch (posStr) {
                case "C": return Position.CATCHER;
                case "P": return Position.PITCHER;
            }
        }
        return null;
    }
}
```

Le convertisseur transforme `Position.CATCHER` en `"C"` dans la base et inversement. Avec `autoApply = true`, il s'applique automatiquement a tous les attributs de type `Position`.

## Exercice 6 : Picture -- Objet embarque (@Embeddable)

**Reponse :**

```java
// Fichier : model/Picture.java (fourni)
package fr.insa.rennes.web.model;

import javax.persistence.Embeddable;

@Embeddable    // Pas une entite : ses champs sont dans la table de l'entite parente
public class Picture {
    private String pic;

    protected Picture() { super(); }

    public Picture(final String img) {
        super();
        pic = img;
    }

    public String getPic() { return pic; }
    public void setPic(final String pic) { this.pic = pic; }
}
```

`@Embeddable` : Picture n'a PAS sa propre table. Ses attributs sont integres dans la table de l'entite qui l'utilise (PlayerCard). On ne peut PAS faire `em.persist(picture)`.

## Exercice 7 : Album -- Relation @OneToMany

### Q7 : Ecrire l'entite avec la relation

**Reponse :**

```java
// Fichier : model/Album.java (fourni)
package fr.insa.rennes.web.model;

import java.util.HashSet;
import java.util.Set;
import javax.persistence.*;

@Entity
public class Album extends ModelElement {
    @OneToMany(
        mappedBy = "album",              // Attribut "album" dans PlayerCard
        cascade = CascadeType.PERSIST,   // Persister Album = persister ses cartes
        fetch = FetchType.LAZY           // Cartes chargees a la demande
    )
    private final Set<PlayerCard> cards;

    public Album() {
        super();
        cards = new HashSet<>();
    }

    public Set<PlayerCard> getCards() { return cards; }
    public void addCard(final PlayerCard pc) { cards.add(pc); }
}
```

Points essentiels :
- `mappedBy = "album"` : la relation est geree par PlayerCard (cle etrangere)
- `CascadeType.PERSIST` : `em.persist(album)` persiste aussi les cartes
- `FetchType.LAZY` : chargement a la demande (les cartes ne sont pas chargees avec l'album)

## Exercice 8 : PlayerCard -- Relations multiples et @Embedded

### Q8 : Ecrire l'entite avec toutes ses relations

**Reponse :**

```java
// Fichier : model/PlayerCard.java (fourni)
package fr.insa.rennes.web.model;

import java.time.LocalDate;
import javax.persistence.*;

@Entity
public class PlayerCard extends ModelElement {
    @ManyToOne
    private Album album;           // Cle etrangere vers Album

    @ManyToOne
    @JoinColumn(name = "PC_PLAYER", nullable = false)
    private Player player;         // Cle etrangere vers Player

    private LocalDate date;

    @Embedded
    @AttributeOverrides({
        @AttributeOverride(name = "pic", column = @Column(name = "FRONT_PIC", nullable = false)),
    })
    private Picture frontPicture;

    @Embedded
    @AttributeOverrides({
        @AttributeOverride(name = "pic", column = @Column(name = "BACK_PIC")),
    })
    private Picture backPicture;

    protected PlayerCard() { super(); }

    public PlayerCard(final Player p, final Picture frontPic, final LocalDate theDate) {
        super();
        player = p;
        frontPicture = frontPic;
        date = theDate;
    }

    public void setAlbum(final Album album) { this.album = album; }
    public Album getAlbum() { return this.album; }
}
```

Schema de la table generee :
```sql noexec
CREATE TABLE PLAYERCARD (
    ID INTEGER NOT NULL,
    DATE DATE,
    FRONT_PIC VARCHAR(255) NOT NULL,
    BACK_PIC VARCHAR(255),
    ALBUM_ID INTEGER,                   -- FK vers ALBUM
    PC_PLAYER INTEGER NOT NULL,         -- FK vers PLAYER
    PRIMARY KEY (ID),
    FOREIGN KEY (ALBUM_ID) REFERENCES ALBUM(ID),
    FOREIGN KEY (PC_PLAYER) REFERENCES PLAYER(ID)
)
```

Points essentiels :
- `@ManyToOne` sur `album` : cote proprietaire (cle etrangere ici)
- `@AttributeOverrides` : renomme les colonnes embarquees pour eviter les conflits (deux Pictures)

public class TestPlayerJPA extends JPATest {

### Q9.4 : Tests Album avec PlayerCards (relation bidirectionnelle)

**Reponse :**

```java
// Fichier : test/TestAlbumJPA.java (fourni)
public class TestAlbumJPA extends JPATest {
    Album album;
    PlayerCard pc1;

    @Override
    void fillDatabase() {
        album = new Album();
        em.getTransaction().begin();
        em.persist(album);
        em.getTransaction().commit();

        Player p1 = new Player("P1");
        em.getTransaction().begin();
        em.persist(p1);
        em.getTransaction().commit();

        Picture pic1 = new Picture("pic/gernot.jpg");
        pc1 = new PlayerCard(p1, pic1, LocalDate.of(2015, Month.JANUARY, 23));
        pc1.setAlbum(album);       // Cote proprietaire
        em.getTransaction().begin();
        em.persist(pc1);
        em.getTransaction().commit();

        album.addCard(pc1);         // Cote inverse (coherence memoire)
    }

    @Test
    void testAlbumWithOnePlayerCard() {
        em.getTransaction().begin();
        final List<Album> albums = em.createQuery("SELECT a FROM Album a", Album.class)
            .getResultList();
        em.getTransaction().commit();

        assertEquals(1, albums.get(0).getCards().size());
        assertEquals(pc1, albums.get(0).getCards().iterator().next());
    }
}
```

Pour une relation bidirectionnelle :
1. `pc1.setAlbum(album)` -- mettre a jour le cote proprietaire (cle etrangere)
2. `album.addCard(pc1)` -- mettre a jour le cote inverse (coherence memoire)

Seul le cote proprietaire (`@ManyToOne`) est ecrit en base.

## Operations essentielles de l'EntityManager

| Methode | Role |
|---------|------|
| `em.persist(entity)` | Inserer en base (INSERT) |
| `em.find(Class, id)` | Rechercher par cle primaire (SELECT) |
| `em.merge(entity)` | Mettre a jour (UPDATE) |
| `em.remove(entity)` | Supprimer (DELETE) |
| `em.createQuery(jpql)` | Executer une requete JPQL |
| `em.createNamedQuery(name)` | Executer une requete predifinie |
| `em.getTransaction().begin()` | Demarrer une transaction |
| `em.getTransaction().commit()` | Valider une transaction |

## Annotations JPA -- Recapitulatif

```
@Entity                  -> table en base
@Id + @GeneratedValue    -> cle primaire auto-generee
@MappedSuperclass        -> superclasse NON persistee

@Inheritance(SINGLE_TABLE | JOINED | TABLE_PER_CLASS)

@OneToMany(mappedBy, cascade, fetch)   -> cote 1 (pas de FK)
@ManyToOne                             -> cote N (FK ici)
@JoinColumn(name = "FK_NOM")           -> nom de la FK

@Embeddable              -> classe embarquee (pas d'entite)
@Embedded                -> utiliser un objet embarque
@AttributeOverrides      -> renommer les colonnes embarquees

@Basic(optional = false) -> NOT NULL
@Column(name, nullable)  -> nom de colonne personnalise
@Transient               -> PAS persiste

@Converter(autoApply)    -> convertisseur de type
@NamedQuery(name, query) -> requete JPQL predifinie
```
