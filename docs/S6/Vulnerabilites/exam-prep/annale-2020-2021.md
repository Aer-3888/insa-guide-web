## Exercice 1 : Qualification d'attaques

### Methode

On presente des lignes de logs web. Pour chaque ligne, identifier le type d'attaque : XSS, SQL injection, command injection, CSRF, ou information gathering.

### Points cles

- Bien regarder la **presence de balises HTML/JS** (XSS) vs **mots-cles SQL** (UNION, SELECT) vs **separateurs shell** (;, |)
- Les requetes d'exploration sans code malicieux sont de l'**INFO** (reconnaissance)
- Les actions sur des parametres sensibles sans token sont du **CSRF**

### Piege classique

Certains logs combinent des elements. Par exemple, une URL avec `id=42 UNION SELECT` ressemble a du SQL, mais si le message d'erreur est `-bash: UNION: command not found`, c'est une injection de commande. Toujours regarder le contexte.

## Exercice 2 : Injection -- SQL ou commande ?

### Enonce type

On presente une application web et une serie de tests :

| Test | Entree | Resultat |
|------|--------|---------|
| 1 | `' OR 'a'='a` | `-bash: OR: command not found` |
| 2 | `admin` / `admin` | Connexion reussie |
| 3 | `' ; /bin/sleep 10` | L'application met 10 secondes |
| 4 | `' ; /bin/cat /etc/shadow` | Permission denied |
| 5 | `admin` / `nawak` | Message d'erreur avec le mot de passe en clair |

### Analyse

**Test 1** : l'erreur vient de **bash**, pas du moteur SQL. Le login est passe au shell. C'est une **injection de commande**, pas SQL.

**Test 3** : le `sleep 10` s'execute (delai de 10s). Confirme l'injection de commande.

**Test 4** : le `cat /etc/shadow` echoue (permissions insuffisantes). L'injection fonctionne mais le processus n'a pas les droits root.

**Test 5** : le message d'erreur affiche le mot de passe en clair. C'est une **fuite d'information** et un probleme de **stockage en clair**.

### Correction proposee

1. Ne pas passer les entrees au shell (utiliser les fonctions natives du langage)
2. Si appel systeme necessaire : whitelist stricte avec regex
3. Stocker les mots de passe en hash sale (bcrypt)
4. Ne jamais afficher les mots de passe dans les messages d'erreur

## Exercice 3 : Securite cloud

### Question type

"En quoi l'environnement cloud peut-il favoriser certains modeles d'attaques ?"

### Elements de reponse

1. **Multitenancy** : partage de ressources physiques cree des canaux auxiliaires
   - Cache CPU (timing attacks)
   - Deduplication de stockage (savoir si un fichier existe)

2. **Confiance deleguee** : le fournisseur a acces a tout
   - Securite physique, confidentialite, disponibilite

3. **Images partagees** : risque de vulnerabilites dans les images publiques
   - Toujours mettre a jour apres deploiement

4. **Surface d'attaque etendue** : interfaces web, API, IAM
   - XSS trouve dans les interfaces AWS

5. **Centralisation** : une compromission affecte de nombreux clients

## Exercice 4 : Protocole cryptographique

### Type de question

On presente un protocole en MSC et on demande :
- Verifier une propriete de securite (secret, authentification)
- Identifier une attaque
- Proposer une correction

### Methode de resolution

```
1. Ecrire les connaissances initiales de chaque role
2. Ecrire les connaissances de l'attaquant (Dolev-Yao : tout ce qui est public)
3. Pour chaque message :
   - L'attaquant peut-il le creer ? (il connait les cles publiques)
   - L'attaquant peut-il le lire ? (il a besoin de la cle privee)
4. Verifier la propriete demandee
5. Si attaque : dessiner le MSC complet avec les deux sessions
```

## Exercice 5 : Mots de passe

### Question type

Un systeme genere des mots de passe avec une formule mathematique. Analyser la securite.

### Elements a verifier

1. **Espace des valeurs** : combien de mots de passe distincts la formule peut-elle generer ?
2. **Principe de Kerckhoffs** : si la formule est connue, les mots de passe sont-ils encore surs ?
3. **Donnees d'entree** : les parametres sont-ils publics (date de naissance) ?
4. **Collisions** : deux utilisateurs peuvent-ils avoir le meme mot de passe ?
5. **Brute force** : combien de temps pour tester toutes les possibilites ?

## Lecons de cette annale

1. **Distinguer SQL et CMD** est un piege classique -- regarder le message d'erreur
2. **Le cloud** est toujours une question ouverte de 2-4 points -- avoir des points cles prepares
3. **Les protocoles** representent la plus grosse partie de la note (~6 pts) -- maitriser la methode MSC
4. **Les mots de passe** combinent mathematiques et raisonnement securite
